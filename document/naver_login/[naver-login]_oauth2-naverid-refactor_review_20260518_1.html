인증 흐름 비교

[ 기존 흐름 ] Naver 로그인 성공 → userInfo.get("id") 를 existsByEmail() 에 넘김 ← id를 email로 쓰는 버그 → Person 저장 시 email = userInfo.get("email") ← email scope 필요 → findByProviderEmail(userInfo.get("id")) ← id를 providerEmail로 쿼리하는 버그 → OAuth2UserEntity 저장 시 providerEmail = email ← email scope 필요 → 프론트에 UserResponse(name, email, birthday, profileImage) 반환 [ 변경 후 흐름 ] Naver 로그인 성공 → naverId = userInfo.get("id") → oAuth2UserRepository.findByProviderId(naverId) 로 존재 여부 단일 체크 → 미존재 시: Person(email=null) + OAuth2UserEntity(providerId=naverId) 저장 → 프론트에 UserResponse(naverId) 반환 — id만 scope에서 제공됨 [ 핵심 변화 ] - 중복 체크 로직을 userRepository + oAuth2UserRepository 두 곳 → oAuth2UserRepository 하나로 통합 - Person.email nullable 허용 (소셜 로그인은 email 없음) - providerEmail 컬럼 완전 제거

의사결정

왜 중복 체크를 oAuth2UserRepository 하나로 통합하는가
기존 코드는 Person과 OAuth2UserEntity를 각각 따로 체크하는데, Person 존재 여부를 email로 체크하는 로직이 id값을 넘기는 버그가 있어 의미가 없었음. OAuth2 로그인 유저는 반드시 OAuth2UserEntity가 있어야 하므로 이를 단일 진실 공급원으로 삼고, Person은 항상 함께 생성.
Person.email을 제거하지 않고 nullable로 두는 이유
추후 일반 이메일/패스워드 로그인 기능을 추가할 경우 email 컬럼이 필요할 수 있음. 현재는 소셜 로그인 유저에 한해 null 허용. DB 스키마 변경 최소화.
UserResponse를 naverId만 반환하도록 변경하는 이유
id scope만 동의받으므로 name, email 등이 Naver로부터 오지 않음. null을 반환하면 프론트에서 혼란 발생. naverId 하나만 명시적으로 반환하고, 프론트에서 로그인 여부 판단에 사용.
provider_email DB 컬럼 처리
ddl-auto: update는 컬럼 삭제를 자동으로 처리하지 않음. 엔티티에서 필드를 제거해도 DB 컬럼은 남음. 서버 반영 후 별도로 ALTER TABLE OAUTH2_USER DROP COLUMN provider_email; 수동 실행 필요. 데이터가 없거나 초기화 가능하면 테이블을 drop & recreate 해도 됨.
