PR 본문 초안

REFACTOR: OAuth2 Naver ID 기반 인증 전환 및 UI 정리

Summary

  • OAuth2 사용자 식별 기준을 이메일에서 Naver 고유 ID로 전환합니다.
  • 기존 코드에 있던 existsByEmail(id값), findByProviderEmail(id값) 버그 2개를 제거합니다.
  • 로그인 화면에서 이메일/비밀번호 폼 및 회원가입 UI를 제거하고 네이버 로그인만 남깁니다.
  • application.yml의 active: prod를 active: local로 변경해 로컬 테스트를 가능하게 합니다.
  • 마이페이지 리뷰 목록에 개수 표시 및 최신순 정렬을 추가합니다.

Changes

  • [Backend] OAuth2UserEntity — providerEmail 필드 제거
  • [Backend] OAuth2UserRepository — findByProviderEmail → findByProviderId
  • [Backend] UserRepository — existsByEmail 제거
  • [Backend] OAuth2UserMapper — email 의존 제거, 시그니처 단순화
  • [Backend] CustomAuthenticationSuccessHandler — 중복 체크를 findByProviderId 단일 조회로 통합
  • [Backend] UserResponse — naverId 단일 필드로 축소
  • [Backend] Controller — isProfileComplete, "/" 엔드포인트 제거
  • [Backend] ReviewController / ReviewService — response.get("email") → response.get("id"), 파라미터명 naverId로 통일
  • [Backend] application.yml — active: local로 변경 (운영서버 SPRING_PROFILES_ACTIVE=prod 필요)
  • [Backend] application-local.yml — scope id 하나로 축소
  • [Frontend] LoginView.vue — 이메일/비밀번호 폼, 회원가입 링크 제거. 네이버 로그인만 유지
  • [Frontend] ReviewCreateView.vue — 작성자 정보 카드 제거, fetchCurrentUser 및 /user/me 호출 제거
  • [Frontend] MyPageView.vue — 리뷰 개수 표시, regDate 기준 최신순 정렬
  • [Frontend] index.html — 타이틀 북토커, lang="ko"로 변경

의사결정

  • 왜 email 대신 Naver ID를 식별자로 쓰는가 — scope를 id만 요청하므로 email을 받을 수 없음. Naver ID는 변경 불가능한 고유값으로 식별에 적합.
  • 중복 체크 로직을 oAuth2Repository 하나로 통합한 이유 — 기존 userRepository.existsByEmail(id값)은 항상 false를 반환하는 버그였음. OAuth2 유저는 반드시 OAuth2UserEntity가 있으므로 이를 단일 진실 공급원으로 사용.
  • application.yml active를 local로 변경한 이유 — 기존 active: prod는 로컬 실행 시에도 prod 설정을 로드해 OAuth2 콜백 URL 불일치로 테스트 불가. 운영서버는 SPRING_PROFILES_ACTIVE=prod 환경변수로 override.
  • 기존 review.writer 데이터 처리 — 기존 writer 컬럼에 이메일 값이 저장돼 있어 TRUNCATE TABLE review CASCADE;로 초기화. 테스트 단계이므로 마이그레이션 불필요로 판단.
