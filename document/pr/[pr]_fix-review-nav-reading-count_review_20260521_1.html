의사결정 과정

이 PR의 핵심은 마이페이지 구조 개편입니다. 단순 버그 수정에서 출발했지만, 수정 과정에서 더 근본적인 UX 문제를 발견하여 구조를 재설계했습니다. 아래는 그 흐름입니다.

문제 발견
버그 2개 — nextReadingCount '?', nav 활성화 안 됨
마이페이지에서 '한번 더 봤어요' 클릭 시 ReviewCreateView의 제목이 항상 '?회독 독후감 작성'으로 표시되는 버그.
독후감 작성 화면에서 상단 nav가 어떤 탭도 활성화되지 않는 버그.
버그 수정
query param 전달 + useNav.js 분기 추가
nextReadingCount: 서버 추가 호출 없이 MyPageView에서 readingCount를 query param으로 전달 → ReviewCreateView에서 +1 computed로 해결.
nav: review-create 라우트에서 mode=next-reading이면 마이페이지 탭, 아니면 검색 탭 활성화.
UX 문제 발견
마이페이지 UI — 모바일 대응 문제
기존 카드 하단에 별점 + '한번 더 봤어요' + '삭제하기' 버튼이 가로 한 줄로 배치되어 모바일(375px)에서 버튼이 줄바꿈되거나 텍스트가 잘림.
같은 책의 1·2·3회독이 각각 별도 카드로 나열되어 목록이 불필요하게 길어지는 문제.
설계 v1 제안
책 단위 그룹핑 + MyBookDetailView 신규 페이지
AI가 설계안을 HTML 문서로 작성하여 검수 요청. 핵심 제안:
· MyPageView를 책 단위 카드로 변경
· 카드 클릭 → MyBookDetailView에서 모든 회독 목록 표시
· 백엔드 변경 없이 프론트에서 /review/list 응답을 isbn13로 그룹핑
검수 피드백
"한 번만 읽은 사람에게 depth가 괜히 추가된다"
v1에서 모든 책 카드 클릭 시 MyBookDetailView로 이동하는 구조가 문제:
회독 기능을 쓰지 않는 사용자(1회독만)는 클릭할 때마다 리뷰가 1개뿐인 중간 페이지를 거쳐야 함 → 불필요한 depth.
설계 v2 수정
클릭 동작을 회독 수에 따라 분기
피드백을 반영하여 조건부 네비게이션으로 수정:
· group.reviews.length === 1 → ReviewDetailView 직행 (기존과 동일한 경험)
· group.reviews.length >= 2 → MyBookDetailView (회독 이력 확인 필요한 사용자만)
추가 API 호출 없이 조건문 하나로 해결.
구현 → 검수 → 수정
구현 후 실제 테스트에서 2가지 추가 문제 발견
· 문제 A: 단독 회독 카드에서 '한번 더 봤어요' 버튼을 제거했더니 2회독을 시작할 진입 경로가 없어짐
→ 단독 회독 카드에 버튼 복원
· 문제 B: MyBookDetailView의 '한번 더 봤어요' 버튼이 flex 컨테이너 내부에 묻혀 레이아웃에 따라 불안정
→ 버튼을 책 제목 영역에서 분리해 독립 행으로 이동
문구 결정
'한번 더 읽기' → '한번 더 봤어요'
구현 중 AI가 '한번 더 읽기'로 문구를 줄여서 작성 → 검수 중 이질감 지적.
"어차피 밑에 인라인이 생겨야만 의미가 있으니 굳이 줄일 이유가 없다"는 판단으로 기존 문구 '한번 더 봤어요' 유지.

주요 의사결정

책 카드에 별점을 표시할까?
A. 최근 회독 별점 표시
카드에 ⭐ N 표시
어떤 회독의 별점인지 모호함
B. 표시 안 함 채택
카드에 별점 없음
다회독 책에서 "어떤 회독의 별점?"이 불명확. 별점은 MyBookDetailView 각 회독 행에서만 표시.
nextReadingCount를 서버에서 매번 조회해야 할까?
A. 서버 API 추가 호출
GET /review/count?isbn13=...
추가 API 엔드포인트 필요, 불필요한 네트워크 비용
B. query param 전달 채택
MyPageView에서 readingCount 전달 → +1 computed
MyPageView가 이미 리뷰 데이터를 보유. 추가 호출 없이 해결 가능.
MyBookDetailView의 정렬 방향은?
A. 오름차순 (1→2→3)
처음 읽은 순서대로
가장 최근 회독이 맨 아래에 위치
B. 내림차순 (3→2→1) 채택
최신 회독이 상단
사용자는 보통 가장 최근 회독이 궁금함. 스크롤 없이 최신 내용 확인 가능.
회독 행 삭제 방식은?
A. 스와이프 삭제 (모바일)
좌→우 스와이프로 삭제 버튼 노출
구현 복잡도 높음, 데스크탑에서 동작 안 함
B. 행마다 🗑 아이콘 채택
각 행 우측에 삭제 버튼 노출
구현 단순, 모바일·데스크탑 모두 동일 UX. 포트폴리오 단계에서 충분.
