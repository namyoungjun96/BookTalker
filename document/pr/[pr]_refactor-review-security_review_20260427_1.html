🔒 Critical Issues Fixed
Critical #1 — insertNextReading 동시 요청 시 DB 오류 메시지 노출
  • 동시 요청으로 unique constraint 위반 시 DataIntegrityViolationException의 DB 원문 메시지가 클라이언트에 그대로 노출되는 문제
  • save()를 try-catch로 감싸 사용자 친화적 메시지로 변환하도록 수정
💬 의사결정 과정
Claude
코드 리뷰 중 발견. check-then-act 패턴의 race condition으로 동시 요청 시 DB 원문 오류가 노출될 수 있음을 지적.
대안 검토: Pessimistic Lock(PESSIMISTIC_WRITE) vs try-catch 래핑
Claude
Pessimistic Lock은 트랜잭션 전체 구간 동안 row를 잠가 성능 부담이 크고, 실제 동시 충돌 빈도가 낮은 이 케이스에는 과도한 처리. try-catch로 DB 원문 메시지를 가로채고 친화적 메시지로 변환하는 방식 제안. DB constraint가 최종 안전망 역할.
결정
try-catch 방식 채택. DataIntegrityViolationException 발생 시 "이미 동일한 회독 독후감이 존재합니다. 잠시 후 다시 시도해주세요." 메시지로 래핑.
Critical #2 — ReviewRequest DTO에 writer 필드 존재 (보안)
  • 클라이언트가 writer 값을 임의로 조작해 타인 명의로 독후감 작성 가능한 취약점
  • writer는 서버가 @AuthenticationPrincipal로 결정하므로 DTO에서 완전 제거
  • 프론트엔드 요청 페이로드에서도 writer 필드 제거
💬 의사결정 과정
Claude
코드 리뷰 중 발견. DTO의 writer 필드를 클라이언트가 임의 값으로 전송 가능한 보안 취약점 지적. 초기 제안: to() 메서드에서 review.setWriter(this.writer)만 제거 — 필드는 유지하되 무시하는 방식.
개발자
더 근본적인 해결책 질문: writer 필드를 DTO에서 아예 제거하는 것이 맞지 않냐고 제안. @AuthenticationPrincipal로 서버가 결정할 값을 DTO에 두는 것 자체가 잘못된 설계 아니냐는 관점.
Claude
동의. DTO는 클라이언트가 제출하는 값만 담아야 하는 계약. writer는 서버 결정값이므로 DTO에 없어야 함. 추가 이점: 서비스 레이어가 SecurityContextHolder에 의존하지 않아 단위 테스트 용이성 유지.
결정
개발자 제안 채택. ReviewRequest DTO에서 writer 필드 완전 제거. 프론트엔드 요청 페이로드에서도 제거. 서비스에서 파라미터로 전달받은 email을 직접 사용.
Critical #3 — AccessDeniedException을 @RestControllerAdvice에서 처리 (401/403 혼재)
  • @RestControllerAdvice에서 AccessDeniedException을 처리하면 Spring Security의 ExceptionTranslationFilter보다 늦게 동작해 401/403 구분이 깨짐
  • AccessDeniedException 핸들러를 GlobalExceptionHandler에서 제거
  • SecurityConfig에 accessDeniedHandler(403) + authenticationEntryPoint(401) 추가로 JSON 응답 처리
💬 의사결정 과정
Claude
코드 리뷰 중 발견. GlobalExceptionHandler에서 AccessDeniedException을 처리하면 Spring Security 필터 체인의 ExceptionTranslationFilter보다 MVC 레이어에서 나중에 잡히는 구조적 문제 지적. 비인증(401)과 인가 실패(403)가 모두 AccessDeniedException으로 처리되어 구분이 깨질 수 있음.
Claude
해결 방향 제안: GlobalExceptionHandler에서 핸들러 제거 후, SecurityConfig의 .exceptionHandling()에 accessDeniedHandler(403 JSON)와 authenticationEntryPoint(401 JSON)를 직접 등록. Spring Security 필터 레벨에서 처리하므로 인증/인가 구분이 정확해짐.
결정
Claude 제안 채택. GlobalExceptionHandler에서 AccessDeniedException 핸들러 제거, SecurityConfig에 401/403 JSON 핸들러 추가.
