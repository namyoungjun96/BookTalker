2 · 의사결정

결정 ①
API 카운팅 단위 — URL 전체 vs Path Pattern
URL 전체 카운팅 시 /reviews/1, /reviews/2가 각각 집계 → Top 3 의미 없음
✗ 비추천
URL 전체
/reviews/123 → 1
/reviews/456 → 1
/reviews/789 → 1
Top 3가 의미 없는 값으로 채워짐
✓ 추천
Path Pattern
/reviews/123 → /reviews/{id} +1
/reviews/456 → /reviews/{id} +1
HandlerMethod의 매핑 패턴 추출로 자동 그룹핑
결정 ②
JVM·DB 정보 수집 — Actuator HTTP endpoint vs Java 내장 API
Actuator HTTP 노출 시 SecurityConfig에서 /actuator/**를 열어야 하고 외부 노출 위험 발생
✗ 비추천
Actuator HTTP endpoint
/actuator/metrics/jvm.memory.used 호출

· SecurityConfig에 /actuator/** permitAll 필요
· 외부 노출 위험
· OCI 1GB RAM에 Actuator 전체 스택 불필요
✓ 추천
Java 내장 API (코드 내부 직접 호출)
JVM: ManagementFactory.getMemoryMXBean()
DB: HikariDataSource → getHikariPoolMXBean()

· 보안 설정 추가 없음
· HTTP 노출 없음
· Actuator 의존성은 추가하되 헬스체크 전용으로만 사용
결정 ③ — 검수 필요
카운팅 대상 — Allowlist vs Blocklist
Interceptor는 모든 요청을 가로채므로 비즈니스 무관 경로는 제외해야 함
A. Allowlist (추천)
명시한 경로만 카운팅
카운팅 대상:
/rank/**
/book/**
/review/**
/user/**

새 API 추가 시 목록에 직접 추가 필요.
카운팅 범위가 명확하고 예측 가능.
B. Blocklist
명시한 경로만 제외
제외 대상:
/login/**, /logout
/oauth2/**
/auth/session
/actuator/**

새 API 추가 시 자동 카운팅.
제외 누락 시 노이즈 데이터 발생 가능.
