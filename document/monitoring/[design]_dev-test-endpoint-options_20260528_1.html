[설계] DevTestController 대안 비교

스케줄러를 즉시 실행해 Discord 리포트를 확인하는 방법 3가지

배경

ApiUsageReportScheduler.sendDailyReport()는 매일 23:59에 실행된다.
개발 중 즉시 실행해 Discord 메시지를 확인하려면 별도 트리거가 필요하다.
현재 구현된 DevTestController(@Profile("local"))를 어떻게 처리할 것인가가 쟁점이다.

대안 비교

A · 현재 방식 @Profile("local") 컨트롤러 유지
장점
  • 브라우저 / Postman에서 즉시 호출 가능
  • prod 환경에서는 빈 자체가 등록 안 됨
단점
  • 테스트 코드가 src/main에 섞임
  • 나중에 지워야 한다는 걸 기억해야 함
  • 코드베이스를 읽는 사람에게 혼란
@RestController
@Profile("local")            // prod에서는 비활성화
public class DevTestController {
    @GetMapping("/dev/report-now")
    public String triggerReport() { ... }
}
B · 추천 Spring Boot 통합 테스트
장점
  • 테스트 코드가 src/test에 명확히 분리
  • DevTestController 파일 자체 불필요
  • 엔드포인트를 SecurityConfig에서 허용할 필요 없음
  • 이후 자동화 테스트로 발전 가능
단점
  • 테스트 실행 시 실제 Discord로 전송됨
  • IDE에서 테스트 실행 필요 (브라우저 호출 불가)
@SpringBootTest
@ActiveProfiles("local")
class ApiUsageReportSchedulerTest {

    @Autowired
    private ApiUsageReportScheduler scheduler;

    @Autowired
    private ApiUsageCounter counter;

    @Test
    void sendDailyReport() {
        // given: 카운터에 데이터 주입
        counter.increment("GET /rank");
        counter.increment("GET /rank");
        counter.increment("GET /book/search");

        // when: 스케줄러 즉시 실행 → Discord 전송
        scheduler.sendDailyReport();
    }
}
C · 절충안 @ConditionalOnProperty 플래그
장점
  • 프로파일이 아닌 프로퍼티로 제어해 더 명시적
  • 필요 시 prod에서도 임시 활성화 가능
단점
  • 테스트 코드가 여전히 src/main에 존재
  • A보다 복잡하면서 B보다 나은 점이 없음
// application-local.yml
app:
  dev:
    enabled: true

// DevTestController.java
@ConditionalOnProperty(name = "app.dev.enabled", havingValue = "true")
public class DevTestController { ... }
결론 (검수 요청)
B (통합 테스트)를 추천합니다.

테스트 코드가 src/test에 분리되고, DevTestController 파일과 SecurityConfig 예외 설정 모두 불필요해져 코드베이스가 깔끔해집니다.

Discord로 실제 전송이 된다는 단점이 있지만, 애초에 리포트 확인이 목적이므로 오히려 자연스럽습니다.
카운터에 더미 데이터를 주입한 뒤 실행하면 메시지 포맷도 정확히 검증됩니다.
