CacheConfig.java

@Configuration @EnableCaching public class CacheConfig { @Bean public CacheManager cacheManager() { CaffeineCacheManager manager = new CaffeineCacheManager("aladin-search"); manager.setCaffeine(Caffeine.newBuilder() .expireAfterWrite(7, TimeUnit.DAYS) .maximumSize(1_000)); return manager; } }

검토 포인트

캐시 키에 maxResults 포함 여부
현재 스펙은 query:start만 키로 사용합니다. 그런데 maxResults가 다르면 (10건 vs 20건) 동일 키로 다른 결과가 나올 수 있습니다.
앱에서 maxResults가 고정값이라면 문제없고, 가변이라면 키에 포함해야 합니다.
→ 확인 필요: 프론트에서 maxResults를 고정으로 보내나요?
list() / searchByIsbn13() 캐싱은 이번 범위 제외
list()(베스트셀러/신간)은 실시간성이 중요해 7일 캐싱이 부적합합니다.
searchByIsbn13()는 이미 DB에 책을 캐싱하는 cacheBook() 흐름이 있어 별도 처리합니다.
→ 이번 브랜치는 search()만 대상으로 합니다.
다중 인스턴스 확장 시 Redis 전환 필요
sync = true는 JVM 내 단일 스레드 진입만 보장합니다. 이미 주석처리된 Redis 의존성이 있는 것으로 보아, 향후 다중 인스턴스 전환 시 캐시 구현체만 교체하면 됩니다. (@Cacheable 애너테이션은 그대로 유지)
