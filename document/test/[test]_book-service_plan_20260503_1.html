1-1. 테스팅 피라미드
🟢 단위 테스트 (Unit Test)
가장 많이 작성 · 빠름 · Mock 사용 · 이 플랜의 핵심
🔵 통합 테스트 (Integration Test)
실제 DB/컨텍스트 연결 · @DataJpaTest / @WebMvcTest
🟠 E2E 테스트
포트폴리오 프로젝트 규모에서는 보통 생략 · 선택 사항
1-2. 레이어별 테스트 필요성 판단
레이어 클래스 테스트 방식 우선순위 판단 근거
Service BookService 단위 테스트 (Mockito) 높음 비즈니스 로직 + 예외 처리 포함. ReviewServiceTest와 동일 패턴 적용 가능
Mapper AladinBookMapper 단위 테스트 (순수 Java) 높음 장르 추출 + 기타 분류 로직이 있어 엣지 케이스 검증 중요. Mock 불필요
Repository BookRepository 통합 테스트 (@DataJpaTest) 낮음 커스텀 쿼리가 없고 JPA 기본 CRUD만 사용. Spring Data JPA가 이미 검증됨
Controller BookController 슬라이스 테스트 (@WebMvcTest) 낮음 HTTP 매핑, 응답 코드 검증용. 포트폴리오 완성도를 위해 추후 추가 가능
Converter CoverSizeEnumConverter
ListQueryTypeConverter		 단위 테스트 (순수 Java) 낮음 단순 Enum 변환. UnknownEnumValueException 발생 조건만 확인하면 충분
💡 지금 추가해야 할 것 — BookService 단위 테스트 + AladinBookMapper 단위 테스트. Repository/Controller/Converter는 커스텀 로직이 거의 없거나 추후 추가해도 충분.
1-3. 단위 테스트를 먼저 작성하는 이유

빠름 실행 속도

  • DB 연결 없이 밀리초 단위
  • CI/CD에서 매 커밋마다 실행 가능
  • Mock으로 외부 API 차단

정밀 로직 검증

  • 예외 경로 강제 테스트 가능
  • 외부 의존성 장애와 무관
  • 엣지 케이스를 직접 주입

재사용 패턴 일관성

  • ReviewServiceTest와 같은 구조
  • @Nested + @DisplayName 패턴
  • 팀 규칙 유지
