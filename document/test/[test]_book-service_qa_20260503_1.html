Question 1 of 2
👤 개발자 질문

getBookByIsbn13WithApi에서 첫 번째 if (response == null) 케이스는 알라딘이랑 통신이 아예 안 된 경우 아닌가? 지금 오류 메시지가 "does not exist"인데 이건 틀린 거 아닌가? 메시지를 바꿔야 할 것 같은데 어떻게 생각해?

🎯 핵심 분석

맞습니다. 두 개의 null/빈값 케이스는 원인이 완전히 다릅니다.

케이스 실제 의미 현재 메시지 적절성
response == null API 통신 자체 실패
네트워크 오류, 타임아웃, 역직렬화 실패 등		 "[BookTalker] ... does not exist" 틀림
item.isEmpty() API는 성공했지만 해당 ISBN 책 없음 "[Aladin] ... does not exist" 맞음

response == null은 책이 없는 것이 아니라 API와 대화 자체가 실패한 상황입니다. 이 상태에서 "does not exist"를 반환하면 호출자가 "책이 없는 것"으로 오해하고 잘못된 처리를 할 수 있습니다.

🔧 제안 수정 방향

메시지 변경만으로도 충분합니다. 예외 타입은 IllegalArgumentException으로 유지합니다 (호출부인 ReviewService가 이 예외를 이미 처리하고 있어 타입 변경 시 영향 범위가 넓습니다).

BookService.java — getBookByIsbn13WithApi()
- throw new IllegalArgumentException("[BookTalker] Book with ISBN13 " + isbn13 + " does not exist");
+ throw new IllegalArgumentException("[Aladin] ISBN13 " + isbn13 + " 조회 중 응답을 받지 못했습니다.");

그리고 테스트 메시지 검증도 함께 수정이 필요합니다:

BookServiceTest.java — 실패_null_응답 케이스
- .hasMessageContaining("[BookTalker]");
+ .hasMessageContaining("[Aladin]")
+ .hasMessageContaining("응답을 받지 못했습니다");

※ 두 번째 케이스(item.isEmpty())의 "[Aladin]" 메시지와 prefix를 통일해 모든 알라딘 관련 오류가 [Aladin]으로 시작하도록 일관성을 유지합니다.

✅
동의. 개발자 직관이 정확합니다. null response는 통신 실패이고, "does not exist"는 명백히 잘못된 메시지입니다. 수정이 필요합니다.
Question 2 of 2
👤 개발자 질문

실패_null_응답 케이스에서 bookRepository가 호출될 수 없는 건 당연한 거 아닌가? 그런데 왜 shouldHaveNoInteractions로 검증하는지 모르겠다.

🎯 핵심 분석

맞습니다. 흐름을 그대로 보면:

getBookByIsbn13WithApi() 실행 흐름 — null 응답 시
1. restTemplate.getForObject() → null 반환
2. if (response == null) → true
3. throw IllegalArgumentException ← 여기서 종료
4. (이하 코드 실행 불가)
5. bookRepository ← 절대 도달 불가

예외로 메서드가 즉시 종료되므로, bookRepository에 접근하는 것은 구조적으로 불가능합니다. 따라서 shouldHaveNoInteractions는 이미 알고 있는 사실을 검증하는 것입니다.

📊 두 관점 비교
관점주장판단
남겨야 한다 미래에 누군가 코드를 변경해 DB를 건드리게 되면 안전망이 됨. "방어적 테스트" 약한 근거
제거해야 한다 현재 코드 구조상 불가능한 경로를 검증하는 것은 노이즈. 테스트의 핵심 의도(예외 메시지 검증)를 흐림. 구현 세부사항(호출 여부)을 테스트하면 리팩터링 시 불필요한 테스트 깨짐 유발 제거 권장

좋은 단위 테스트는 행동(behavior)을 검증합니다. "이 입력에 이 예외가 나야 한다"가 이 테스트의 의도이고, 그 이상의 구현 경로 검증은 테스트를 취약하게 만들 수 있습니다.

BookServiceTest.java — 실패_null_응답 (권장 형태)
assertThatThrownBy(() -> bookService.getBookByIsbn13WithApi(ISBN13))
.isInstanceOf(IllegalArgumentException.class)
.hasMessageContaining("[Aladin]")
.hasMessageContaining("응답을 받지 못했습니다");
-
- then(bookRepository).shouldHaveNoInteractions();
✅
동의. 개발자 직관이 맞습니다. shouldHaveNoInteractions는 이 케이스에서 불필요한 검증입니다. 핵심 행동(예외 발생 + 메시지)만 검증하는 것이 더 깔끔한 테스트입니다.
