ReviewRequest.to()에서 review.setWriter(this.writer)를 제거했습니다. writer는 이제 서비스 레이어에서 @AuthenticationPrincipal OAuth2User로부터만 설정됩니다. 프론트에서도 writer 필드 전송 중단 + currentUser null 시 버튼 비활성화.
U
사용자 의견
더 근본적인 해결책으로는 AuthenticationPrincipal에서 이름 정보를 꺼내는 것인 것 같은데 어떻게 생각하는지? — writer 필드를 DTO에서 아예 없애는 방향이 맞지 않냐는 취지.
C
Claude 의견
두 가지 방향을 분석했습니다:

① SecurityContextHolder를 서비스에서 직접 사용
writer 파라미터 자체가 사라져 위조 여지가 근본적으로 없어지지만, 서비스가 Spring Security 인프라에 강하게 결합됩니다. 테스트마다 SecurityContextHolder 세팅이 필요해져 기존 16개 테스트 모두 영향.

② ReviewRequest에서 writer 필드 자체를 제거
컨트롤러→서비스로 email 문자열을 넘기는 현재 패턴을 유지하되, DTO에서 writer가 사라지므로 "클라이언트가 보내도 되는 값"처럼 보이는 스멜이 완전 제거. 서비스 테스트 가능성도 유지됨. 이것이 더 올바른 설계라고 판단.
U
사용자 결정
음 좋아 그 방향이 맞다. 테스트 코드도 그에 맞게 고쳐줘. — ② DTO에서 writer 필드 완전 제거 방향으로 확정.
결론 및 설계 원칙 ReviewRequest는 클라이언트가 제출하는 값만 담아야 합니다. writer(작성자)는 클라이언트가 제어하는 값이 아니라 서버가 인증 컨텍스트에서 결정하는 값입니다. DTO 시그니처에 필드가 있다는 것 자체가 "이 값을 보내도 된다"는 암묵적 신호이므로, 필드를 제거하는 것이 문서화이자 보안 설계입니다. 컨트롤러 → 서비스로 String email을 파라미터로 넘기는 패턴은 표준이며, 서비스를 순수 Java로 유지하고 단위 테스트 가능성을 보장합니다.
