#1 Race Condition — insertNextReading에서 readingCount 충돌 가능
📁 ReviewService.java · insertNextReading()

findMaxReadingCount() 조회 후 save()까지 사이에 락이 없습니다. 동일 사용자가 동시에 두 번 요청하면 두 요청 모두 같은 maxReadingCount를 읽고, 동일한 readingCount로 저장을 시도합니다. DB 유니크 제약이 잡아주긴 하지만, 이때 노출되는 에러 메시지는 Hibernate 내부 메시지(DataIntegrityViolationException)가 그대로 반환됩니다.

권장 수정 insertNextReading 내부에서 save()를 try-catch로 감싸 DataIntegrityViolationException을 잡아서 사용자 친화적 메시지의 DataIntegrityViolationException으로 재던지거나, SELECT ... FOR UPDATE(비관적 락)을 적용합니다.
#2 writer 필드 보안 스멜 + 프론트에서 'anonymous' 폴백
📁 ReviewRequest.java · ReviewCreateView.vue:278

ReviewRequest.to()가 클라이언트 요청의 writer로 엔티티를 설정하고 서비스가 즉시 덮어씁니다. 덮어쓰는 코드가 빠지면 권한 상승 버그가 됩니다.

프론트엔드에서는 currentUser가 null이면 'anonymous'를 전송합니다. 세션 만료 시 사용자에게 명확한 오류 대신 Spring Security의 302/401이 반환됩니다.

권장 수정 ReviewRequest.to()에서 review.setWriter(this.writer)를 제거합니다. 프론트에서는 currentUser가 null이면 폼 제출을 막고 재로그인 안내를 표시합니다.
#3 GlobalExceptionHandler가 Spring Security의 AccessDeniedException을 가로챔
📁 GlobalExceptionHandler.java

AccessDeniedException은 Spring Security의 ExceptionTranslationFilter가 처리하는 예외입니다. @RestControllerAdvice가 이를 가로채면 미인증(401)과 인가 실패(403)를 구분하지 못하고, Spring Security의 세션/리다이렉트 흐름도 끊깁니다.

권장 수정 SecurityConfig에 .exceptionHandling(e -> e.accessDeniedHandler(...).authenticationEntryPoint(...))를 설정해 JSON 응답을 반환하도록 하고, GlobalExceptionHandler의 AccessDeniedException 핸들러는 제거합니다. 또는 현재 방식을 유지하되 주석으로 의도를 명시합니다.
▲ Important — 수정 권장
#4 같은 책의 모든 회독 카드에 "한 번 더 봤어요" 버튼이 표시됨
📁 MyPageView.vue

동일 ISBN의 1회독, 2회독 카드가 모두 존재하면 두 카드 모두에 "한 번 더 봤어요"가 보입니다. 기능적으로는 무관하지만(백엔드가 max+1을 계산하므로), 사용자에게 혼란스럽습니다.

권장 수정 ISBN13별로 그룹핑 후 가장 높은 readingCount인 카드에만 버튼을 표시합니다.
#5 currentUser가 null이면 폼 제출이 차단되지 않음
📁 ReviewCreateView.vue:278

네트워크 오류나 세션 만료로 fetchCurrentUser가 실패해도 폼을 제출할 수 있습니다. 요청은 Spring Security에 의해 거부되지만 사용자는 이유를 알 수 없습니다.

권장 수정 currentUser가 null이면 등록 버튼을 비활성화하고 "로그인 상태를 확인해주세요" 안내를 표시합니다.
#6 insertNextReading에 입력값 유효성 검증 없음
📁 ReviewService.java · insertNextReading()

updateReview는 headline, content, rating을 검증하지만 insertNextReading에는 동일한 검증이 없습니다. rating이 null이면 DB에 null이 저장되거나 NPE가 발생해 500 오류가 반환됩니다.

권장 수정 공통 private 검증 메서드를 추출해 insertReview, insertNextReading, updateReview 모두에서 호출합니다.
#7 @Setter 클래스 레벨 — id와 readingCount에도 public setter 생성됨
📁 Review.java

@Setter가 클래스 전체에 적용되어 setId()와 setReadingCount()가 모두 공개됩니다. review.setId(otherId)를 호출하면 JPA가 insert 대신 merge를 수행하여 기존 데이터를 덮어쓸 수 있습니다.

권장 수정 id와 readingCount에는 @Setter를 적용하지 않고 필드 레벨로 나머지에만 붙이거나, @Builder로 전환합니다.
#8 Integer readingCount — null 가능 타입이 nullable=false와 충돌
📁 Review.java

Integer(박싱 타입)는 Java에서 null 가능합니다. setReadingCount()가 누락된 코드 경로가 생기면 Hibernate가 null을 DB에 전달하거나 NPE가 발생합니다.

권장 수정 private int readingCount; 프리미티브로 변경합니다. null 자체가 불가능해져 방어 코드가 불필요해집니다.

#9 회독 독후감 작성 페이지 제목이 항상 "?회독"으로 표시됨
📁 ReviewCreateView.vue · nextReadingCount

nextReadingCount가 ref('?')로 초기화되고 업데이트되지 않아 항상 "?회독 독후감 작성"으로 표시됩니다.

권장 수정 onNextReading()에서 query.readingCount = review.readingCount + 1을 함께 전달하고, ReviewCreateView에서 route.query.readingCount로 읽습니다.
#10 POST 성공 응답이 200 OK — REST 관례상 201 Created 권장
📁 ReviewController.java

리소스 생성 성공 시 HTTP 표준은 201 Created입니다. 현재 POST /api/review와 POST /api/review/next-reading 모두 200 OK를 반환합니다.

권장 수정 return new ResponseEntity<>(HttpStatus.CREATED);로 변경합니다. 프론트엔드는 2xx이면 성공으로 처리하므로 별도 수정 없이 동작합니다.
#11 공개 리뷰 조회 시 엔티티 전체 fetch — 프로젝션 최적화 가능
📁 ReviewRepository.java · findPublicReviews

PublicReviewResponse는 5개 필드만 필요하지만 Review 엔티티 전체를 조회합니다. Book JOIN까지 포함되므로 페이지당 쿼리 비용이 불필요하게 높습니다.

권장 수정 JPQL Constructor Expression 또는 Spring Data Projection Interface를 사용해 필요한 필드만 조회합니다.
#12 ReviewServiceTest — 성공 케이스에서 getBook() 단언 누락
📁 ReviewServiceTest.java · 성공_DB에_책_존재

DB에 책이 있을 때의 성공 케이스에서 saved.getBook() 단언이 없습니다. setBook()이 누락돼도 테스트가 통과합니다.

권장 수정 assertThat(saved.getBook()).isEqualTo(existingBook);을 추가합니다.
#13 nextReadingId — 네비게이션 실패 시 버튼 "이동 중..." 상태 고착
📁 MyPageView.vue · onNextReading()

router.push()가 await 없이 호출됩니다. 네비게이션 가드 등으로 이동이 취소되면 nextReadingId가 리셋되지 않아 버튼이 영구적으로 "이동 중..." 상태로 남습니다.

권장 수정 await router.push(...) 후 finally 블록에서 nextReadingId.value = null로 리셋합니다.
