Infrastructure · Deployment

Blue-Green 무중단 배포
설계 및 구현

OCI Free Tier 단일 서버 환경에서 심볼릭 링크 기반 Blue-Green 배포를 직접 설계하고 구현한 과정

환경 OCI Free Tier · 1GB RAM
스택 GitHub Actions · Nginx · Shell Script
프로젝트 BookTalker
Background

왜 무중단 배포가 필요했나

독서 동아리 실사용자가 생기면서 배포 시 서비스가 중단되는 문제가 발생했다. 기존 방식은 JAR을 교체하고 프로세스를 재시작하는 단순 배포였는데, Spring Boot 기동 시간이 최대 1분이 걸리는 OCI Free Tier 환경에서는 사용자가 그대로 에러를 마주치는 상황이었다.

Key Decisions

핵심 의사결정

배포 전략 선택 — Rolling / Blue-Green / Canary 중 무엇을?
Blue-Green 선택
Rolling과 Canary는 여러 인스턴스가 필요하다. OCI Free Tier는 서버가 1대라 스케일 아웃이 불가능하므로 단일 서버에서 포트를 나눠 두 인스턴스를 운영하는 Blue-Green만 현실적인 선택지였다.
JAR 관리 방식 — 파일 교체 vs 심볼릭 링크?
심볼릭 링크 + versions/ 폴더로 버저닝
매번 같은 파일명으로 덮어쓰면 롤백이 불가능하다. Git commit hash로 버저닝된 JAR을 versions/ 폴더에 보관하고, book_talker_backend.jar 심볼릭 링크만 전환하는 방식으로 롤백 구조를 단순화했다. Nginx conf 전환도 동일한 심볼릭 링크 패턴을 사용해 일관성을 확보했다.
헬스체크 기준 — 1회 성공 vs 연속 N회?
10초 간격 연속 3회 성공
1회 성공은 JVM 워밍업이나 DB 커넥션 풀 안정화 전에 트리거될 수 있다. 10초 간격으로 3번 연속 200 응답을 확인해야 안정적으로 뜬 것으로 판단하도록 했다. OCI에서 기동 시간이 최대 1분이라 사전 sleep 70초를 추가했다.
배포 실패 시 롤백 전략 — 자동 vs 수동?
헬스체크 실패 시 자동 종료 + Discord 알림
헬스체크가 3번 모두 실패하면 새 인스턴스를 종료하고 Discord Webhook으로 알림을 보낸다. Nginx는 기존 포트를 가리키고 있는 상태이므로 별도 롤백 없이 자연스럽게 기존 버전이 서빙된다. 복잡한 자동 롤백 로직보다 단순하고 안전하다.
환경변수 관리 — /etc/bash.bashrc vs /etc/environment?
/etc/environment 사용
/etc/bash.bashrc는 interactive 쉘에서만 로드된다. GitHub Actions에서 SSH로 실행하는 non-interactive 쉘 환경에서는 환경변수가 전달되지 않는 문제가 있었다. /etc/environment는 쉘 종류와 무관하게 로드되므로 이 방식으로 변경했다.

Deployment Flow

배포 흐름

01
현재 운영 포트 판단
Nginx 심볼릭 링크를 readlink로 확인해 현재 Blue/Green 중 어느 쪽이 서빙 중인지 판단. 반대 포트를 NEXT_PORT로 지정.
readlink /etc/nginx/sites-enabled/booktalker
02
JAR 심볼릭 링크 전환
versions/ 폴더의 최신 JAR을 book_talker_backend.jar 심볼릭 링크로 연결.
ln -sf versions/backend-{hash}.jar book_talker_backend.jar
03
새 인스턴스 실행
NEXT_PORT로 Spring Boot 실행. OCI 기동 시간을 고려해 70초 대기 후 헬스체크 시작.
sh backend.sh start {NEXT_PORT} → sleep 70
04
헬스체크 3회 연속 확인
10초 간격으로 /actuator/health에 200 응답이 연속 3회 오면 안정적으로 기동된 것으로 판단.
curl http://localhost:{NEXT_PORT}/actuator/health × 3
05
Nginx 전환 + reload
sites-enabled 심볼릭 링크를 새 conf로 전환. nginx -t로 문법 검증 후 reload. restart가 아닌 reload라 기존 연결이 끊기지 않는다.
ln -sf booktalker-green booktalker → nginx -t → systemctl reload nginx
06
기존 인스턴스 종료
전환 완료 후 기존 포트 프로세스를 graceful stop.
sh backend.sh stop {CURRENT_PORT}
✕
헬스체크 실패 시 롤백
3회 중 실패가 있으면 새 인스턴스를 종료하고 Discord로 알림 전송. Nginx는 그대로이므로 기존 버전이 자동으로 서빙됨.
kill {NEXT_PORT PID} → Discord Webhook 알림 → exit
Outcome

결과 및 트레이드오프

달성한 것

배포 시 서비스 중단 없음. Nginx reload가 기존 커넥션을 유지한 채 포트를 전환.

현실적 제약

1GB RAM에서 두 인스턴스가 동시에 뜨는 약 1-2분 동안 메모리 80% 점유. 허용 가능한 수준으로 판단.

롤백 구조

versions/ 폴더에 commit hash 기반 JAR 보관. 심볼릭 링크만 바꾸면 언제든 이전 버전으로 복귀 가능.

운영 가시성

배포 성공/실패를 Discord Webhook으로 즉시 알림. 장애 발생 시 빠른 대응 가능.
