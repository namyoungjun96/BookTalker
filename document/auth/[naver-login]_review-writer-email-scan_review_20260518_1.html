[검수] Review writer 이메일 → Naver ID 전환

스캔 결과 — 이메일 의존 코드

파일 라인 문제
ReviewController.java 38, 48, 58, 69, 77, 86 BUG response.get("email") — email scope 제거 후 항상 null 반환. writer=null로 저장되거나 NullPointerException 발생 (6개 엔드포인트 전부)
ReviewService.java 81, 116, 125, 140 수정 파라미터명이 email로 된 메서드 4개. 로직은 동일하지만 naverId로 네이밍 정리 필요. getReviews 로그 메시지도 "email" 문자열 포함
Review.writer DB 컬럼 — 데이터 주의 현재 writer에 이메일 값이 저장돼 있음. 변경 후 Naver ID로 저장되므로 기존 데이터와 불일치 발생 (아래 별도 섹션 참고)

데이터 마이그레이션 이슈

⚠ 기존 review.writer 데이터 불일치
현재 REVIEW.writer 컬럼에는 이메일 주소가 저장돼 있음.
변경 후에는 Naver ID(숫자 문자열)로 저장되므로, 기존 행은 어떤 사용자와도 매칭되지 않음.

선택지:
1. 리뷰 데이터 초기화 — TRUNCATE TABLE review CASCADE; (테스트/포트폴리오 단계라면 가장 단순)
2. 그냥 둠 — 기존 데이터는 조회 안 되지만 서비스 자체는 정상 동작. 새로 쓰는 독후감부터 Naver ID로 정상 저장됨

※ OAuth2UserEntity에서 providerEmail을 이미 제거했기 때문에 email → Naver ID 매핑 쿼리로 마이그레이션하는 방법은 불가.

변경 파일 요약

파일변경 내용
ReviewController.java 6개 메서드 전부: response.get("email") → response.get("id"), 변수명 email → naverId
ReviewService.java 파라미터명 email → naverId (4개 메서드), 로그 문자열 정리. 로직 변경 없음

Review.java, ReviewRepository.java, ReviewRequest.java — writer 필드명 자체는 이메일/ID 무관하게 동작하므로 변경 불필요

의사결정

Review.writer 컬럼명은 그대로 유지하는 이유
writer는 "작성자 식별자"라는 의미로 여전히 유효. 내부 저장값이 email에서 naverId로 바뀌는 것이지 컬럼의 역할은 동일. DB 컬럼명 변경은 별도 마이그레이션이 필요해 비용 대비 이득 없음.
기존 리뷰 데이터 처리 방향 결정 필요
테스트 단계이면 TRUNCATE TABLE review CASCADE;로 초기화. 실데이터가 있다면 그냥 두는 것도 허용 — 새 독후감부터 정상 동작하므로 서비스 자체는 문제 없음.
