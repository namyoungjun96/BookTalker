BookTalker · 결정 기록 05

배치 구현과 통합 테스트

설계만 해뒀던 배치 본체를 처음으로 끝까지 구현했고(가드부터 Top10 컷까지), 그것을 실제 DB로 검증하는 통합 테스트를 세웠다. 그리고 "테스트에 랜덤을 쓰면 안 되는 이유"를 직접 부딪혀 정리했다.

이번 세션 · 완료
  • 배치 본체 구현 완료 — 가드 → C → 가중평점 → 정렬 → 장르별 Top10 → deleteAll/saveAll
  • m을 상수로 (CONFIDENCE_THRESHOLD) · 호출부에서 넘기고 계산기는 파라미터로 받음
  • 정렬·그룹핑·Top10을 stream으로 — sorted().reversed() → groupingBy → flatMap(limit)
  • 통합 테스트 "생성·갈아엎기" — 빈 결과 가드 테스트의 짝
  • 랜덤 rating 제거 — 결정적 테스트 원칙 정리
01

배치 본체 — 흐름 1~7 완성

설계(결정 기록 03)대로, 빠졌던 정렬·Top10까지 채움

집계 결과가 비면 로그 + 종료
deleteAll보다 앞 · 기존 랭킹 보호
C 계산
calculateGlobalAverage(rawData) · 한 번만
각 책 가중 평점 → Rank로 변환
calculateWeightedScore(reviewCount, avgRating, CONFIDENCE_THRESHOLD, globalAverage)
가중 평점 내림차순 정렬이번에 추가
sorted(comparing(getWeightedScore).reversed()) · 그룹핑보다 먼저
장르별 그룹핑 → 각 Top10이번에 추가
groupingBy(genre) → flatMap(list.stream().limit(10))
deleteAll → saveAll
@Transactional로 원자성 보장
if (rawData.isEmpty()) { log.info(...); return; }
double globalAverage = WeightedScoreCalculator.calculateGlobalAverage(rawData);

Map<String, List<Rank>> grouped = rawData.stream()
    .map(row -> { /* Rank 생성 + setWeightedScore(...) */ })
    .sorted(Comparator.comparing(Rank::getWeightedScore).reversed())
    .collect(Collectors.groupingBy(Rank::getGenre));

List<Rank> ranks = grouped.values().stream()
    .flatMap(list -> list.stream().limit(10))
    .collect(Collectors.toList());

rankRepository.deleteAll();
rankRepository.saveAll(ranks);
limit 위치
limit(10)은 그룹핑 후, 각 그룹 안에서여야 한다. 정렬 직후 전체에 limit을 걸면 "전체 상위 10개"가 되어 한 장르가 통째로 빠질 수 있다 — 원하는 건 "장르마다 각각 10개". 그리고 정렬이 그룹핑보다 먼저여야 각 그룹이 정렬된 상태를 유지한다.
stream vs for

일관성도 가독성이다

"각 장르 10개 뽑아 합치기"를 for문으로 풀 수도 있었지만, 위에서 stream을 쓰다 아래서 for로 바꾸면 패러다임이 섞여 더 헷갈린다. flatMap이 충분히 읽힌다고 판단해 stream으로 통일. (도구를 맹목적으로가 아니라 일관성 기준으로 고른 것 — 면접 포인트.)

02

통합 테스트 · 생성과 갈아엎기

결정 기록 04의 "반쪽 가드 테스트"를 완성하는 짝

가드 테스트는 "빈 결과 → 기존 보존"만 봤다(반쪽). 이 테스트는 "데이터 있음 → 기존 지우고 새로 생성"을 본다. 둘이 합쳐져 deleteAll이 "데이터 있을 때만" 작동함이 증명된다. if(true) return; 같은 "항상 종료" 버그도 이 테스트가 잡는다.

핵심 트릭

기존과 새 것의 isbn을 다르게

given의 기존 Rank(예: ...1011)와 오늘 집계될 책(예: ...1112)을 다른 isbn으로 둔다. 배치 후 hasSize(1) + "남은 게 1112"를 확인하면 — 기존 1011이 사라졌고(deleteAll) 새 1112가 생긴 것(saveAll)이 한 번에 증명된다. 기존이 안 지워졌으면 size가 2일 테니.

데이터 정리
@Transactional을 안 쓰는 통합 테스트라 자동 롤백이 없다. @AfterEach에서 review → book 순으로 deleteAll(외래키 때문에 자식 먼저). rank도 함께. 안 그러면 다음 테스트가 오염된다.
03

테스트에 랜덤을 쓰면 안 된다

"다양한 값이 필요하다"는 직감은 맞았지만, 랜덤은 그 답이 아니다

✗ 랜덤 rating

random.nextInt(5)

매번 값이 달라져 같은 입력 → 같은 결과(결정적)가 깨진다. 어떤 날 통과, 어떤 날 실패하는 flaky 테스트. 평균이 매번 달라 기대값을 못 적는다 — 정렬·점수 검증이 불가능해진다. (게다가 nextInt(5)는 0~4라 0점이 섞임.)

✓ 의도된 고정값

책 A=5점들, B=4점들, C=3점들

결과 순서가 확정된다 → then에서 "A, B, C 순으로 저장됐다"를 단언할 수 있다. 테스트 데이터는 결과를 예측할 수 있게 내가 설계한다.

"다양한 값"이 필요한 게 아니라 "서로 다른, 내가 아는 값"이 필요했다. 랜덤은 예측 불가라 정반대 도구다. 테스트가 보려던 건 "배치가 집계를 제대로 하나"이지 "무작위 입력에 견디나"가 아니었다.
04

다음 · 남은 배치 테스트

rating을 의도적으로 설계해 결과를 예측 가능하게 만드는 게 관건

  • 가중 평점 정렬 검증 — rating·리뷰 개수를 설계해 순서를 확정하고, 저장된 Rank가 그 순서인지shrinkage까지: 5.0(리뷰 3개) < 4.5(리뷰 10개)가 나오면 가중평점이 진짜 작동하는 증거 · 면접에서 제일 보여줄 테스트
  • 장르별 Top10 컷 + 장르 분리 — 장르 2개 × 각 11권, 각 장르 10개씩만 저장되나상수를 낮추지 말고 진짜 11권으로 — 테스트와 운영이 같은 값으로 돌아야 함
  • weightedScore 값 정확성 — 저장된 점수가 계산식과 일치, avgRating은 표시용으로 따로 저장됐나
현재 손에 있는 것
배치 본체 구현 완료 · 쿼리 단 테스트(리뷰3개·회독수) · 계산 단 테스트(가중평점·C·예외) · 통합 테스트(가드 / 생성·갈아엎기). 다음은 정렬·Top10을 통합 테스트로 — 헬퍼는 책마다 다른 rating·개수를 줄 수 있게 유연하게.
