가중 평점 랭킹
남은 작업 로드맵

계산 로직(가중 평점 · C)은 단위 테스트까지 끝냈다. 이제 그 계산을 실제 배치 흐름에 엮고, 쿼리·엔티티를 맞추고, 흐름 전체를 테스트하는 일이 남았다. 순서대로 무엇을·왜·어떻게 검증하는지 정리했다.

현재 상태 · 완료
  • calculateWeightedScore — 책 한 권 가중 평점 · 단위 테스트 2개(정상 / shrinkage)
  • calculateGlobalAverage — C(리뷰 단위 전체 평균) · 단위 테스트 2개(정상 / 빈 리스트 예외)
  • BookRatingStats — 집계 재료 record(불변)
  • 방향 확정 — 집계는 DB, 계산·정렬·Top10은 배치에서 / Rank엔 avgRating(표시) + weightedScore(정렬)
01

남은 작업 순서

아래로 갈수록 기존 코드에 손이 커진다 — 테스트가 안전망이 되어주는 순서

다음 차례

Rank 엔티티에 weightedScore 컬럼 추가

무엇
avgRating(단순 평균, 화면 표시용)은 그대로 두고 weightedScore(가중 평점, 정렬용)를 추가.
왜
사용자에겐 익숙한 단순 평균을 보여주고 순위만 가중 평점으로 매기기로 했으니 둘 다 저장해야 한다.
검증
엔티티 변경만이라 별도 테스트는 작음. 마이그레이션/컬럼 추가 확인 정도.
DB 레이어

집계 쿼리: Object[] → BookRatingStats 생성자 표현식

무엇
SELECT new ...BookRatingStats(...) 형태로 바꿔 의미 있는 객체로 받기. 정렬(ORDER BY AVG)은 제거 — 어차피 자바에서 가중 평점으로 다시 정렬.
주의
생성자 파라미터 순서를 SELECT 순서와 정확히 맞춰야 함. 패키지 경로 풀로 적기. 회독1·HAVING COUNT>=3 조건은 그대로.
검증
@DataJpaTest — 실제 데이터 넣고 쿼리 실행해서 1회독만/리뷰 3개 경계/장르 분리가 진짜 맞는지. (Mockito로는 못 잡는다 — 쿼리 자체가 검증 대상)
조립

배치(aggregateRank)에서 전체 흐름 엮기

집계 쿼리 → List<BookRatingStats>
if (비어있으면) { 로그 + return; }   // 정상 종료
C = calculateGlobalAverage(목록)
각 책: weightedScore = calculateWeightedScore(v, R, m, C)
가중 평점 내림차순 정렬 → 장르별 Top10 → Rank 변환 → 저장
왜
계산기 두 개는 독립적으로 테스트했고, 여기서 비로소 "재료 → 완제품(Rank)" 변환과 조율이 일어난다. Top10 자르기도 여기(집계 시점)로.
검증
배치 흐름 테스트 — Top10 컷, 가중 평점 정렬, 장르 안 섞임, 빈 결과면 저장 안 함. (행동 목록의 남은 항목 대부분이 여기)
조회 정리

getRank를 순수 읽기로 단순화

무엇
조회 시점의 findTop10By... 계산 제거. 테이블에 이미 장르별 10개씩만 들어 있으니 장르 필터해서 weightedScore 순으로 꺼내기만.
왜
계산 0, 자르기 0 — Rank 테이블을 캐시로 둔 취지 회복(DR-01).
순서의 의미
위에서 아래로 갈수록 기존 코드 변경 폭이 커진다. 그래도 각 단계마다 테스트가 먼저/함께 서기 때문에 "옮겨도 결과 같다"가 보장된 채로 안전하게 진행된다. 이게 테스트를 안전망 삼은 리팩토링.
02

이번에 새로 만날 것들

남은 단계에서 처음 부딪힐 개념 — 미리 알고 가면 덜 막힌다

JPQL 생성자 표현식 — 쿼리 결과를 DTO 생성자로 바로 매핑. GROUP BY·AVG·COUNT 집계엔 인터페이스 Projection보다 이게 잘 맞는다. 잔손(경로·순서)이 있어 처음엔 헷갈릴 수 있음.

@DataJpaTest — 단위 테스트와 달리 실제 DB(보통 인메모리 또는 테스트 컨테이너)에 데이터를 넣고 쿼리를 돌린다. given이 "객체 몇 개"가 아니라 "DB에 행 삽입"이라 손이 더 간다. 쿼리에 든 로직(회독수·필터)은 이걸로만 검증된다.

배치 흐름 테스트 — 계산기 단위 테스트와 달리, 집계→계산→정렬→저장이 엮인 걸 본다. 어디까지 실제로 돌리고 어디를 가짜로 둘지(예: 쿼리 결과를 미리 준 목록으로 대체) 판단이 필요하다.

03

그 너머 (서두르지 않기)

랭킹이 일단락된 뒤, 데이터·피드백을 보고 결정할 것들

  • Rank 도메인 분리 — review 패키지에서 rank로 독립 검토방향은 분리가 맞음(다른 생명주기·변경 이유). 집계 쿼리 소속이 회색지대라 차분히 결정
  • 변수·이름·예외 메시지 다듬기테스트 이름은 "상황 + 기대결과"까지 / 예외엔 이유 메시지 / 매직 넘버 상수화
  • 캐싱(Caffeine) 도입 검토사용자·트래픽 데이터 확인 후. 지금은 과함
  • 장르별 C 재검토데이터 쌓이고 장르별 평점 분포가 갈리면
데이터가 적은 지금, 가중 평점의 실효는 거의 없다. 하지만 이건 "지금 더 나은 랭킹"이 아니라 "데이터가 쌓였을 때 올바르게 동작하도록 한 설계"다. 불확실한 미래 대비는 YAGNI지만, 데이터가 쌓이는 건 확정된 방향이라 이건 정당한 대비다.
우선순위
결정의 무게를 결정의 영향에 맞춘다. 영향 큰 결정(가중 평점 도입, C 정의)엔 시간을 쓰고, 영향이 0에 수렴하는 결정(static이냐 빈이냐 같은)엔 빨리 고르고 넘어간다 — 그것도 엔지니어링 능력.
