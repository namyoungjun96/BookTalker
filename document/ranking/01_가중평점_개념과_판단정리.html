가중 평점 랭킹
개념과 판단 정리

랭킹 산정 로직을 단순 평균에서 가중 평점으로 바꾸면서 내린 설계 판단과, 그 과정에서 다시 잡은 테스트 개념을 한 곳에 모았다. 코드를 어떻게 짰는지가 아니라 왜 그렇게 정했는지를 복습하기 위한 문서.

대상 BookTalker · RankService / WeightedScoreCalculator 용도 면접 대비 · 판단 근거 복습
01

핵심 개념 · 가중 평점

표본이 적은 평균을 왜 못 믿는가, 그리고 어떻게 보정하는가

리뷰 3개짜리 책이 우연히 5.0을 받았다고, 리뷰 50개에 평균 4.5인 책을 제치는 건 직관적으로 이상하다. 표본이 적으면 평균의 분산이 커서 그 값을 믿기 어렵기 때문이다. 3명이 5점을 준 건 운일 수 있지만 50명이 4.5를 준 건 거의 확실하다.

해결의 핵심 아이디어: 리뷰가 적은 책의 평점은 전체 평균 쪽으로 끌어당긴다. 리뷰가 쌓일수록 그 책 자신의 평균을 점점 더 신뢰하고, 적으면 "아직 못 믿으니 일단 평타(전체 평균) 취급"한다. 이 보정을 shrinkage(축소)라고 부른다.

WR = ( v / (v + m) ) × R  +  ( m / (v + m) ) × C R — 그 책의 평균 별점  ·  v — 그 책의 리뷰 수
m — 신뢰를 인정하는 기준 리뷰 수 (BookTalker는 3)  ·  C — 전체 평균 별점

직관만 잡으면 된다. v가 크면 v/(v+m)이 1에 가까워져 자기 평균 R을 거의 그대로 쓰고, v가 작으면 C(전체 평균) 비중이 커져 평타로 끌려온다. m이 그 무게추다.

한계 인지
초기엔 책·리뷰가 적어 C가 흔들릴 수 있다. 이 한계까지 알고 있다는 것 자체가 면접에선 오히려 좋은 이야깃거리다. (이진 평가였다면 Wilson score가 더 맞지만, 별점이라 위 가중 평점이 자연스럽다.)
02

C(전체 평균) 정책

기술 문제가 아니라 "기준선이 무엇을 대표하는가"라는 도메인 결정

리뷰 단위 vs 책 단위

같은 데이터라도 C를 구하는 방식에 따라 값이 갈린다. 차이의 본질은 리뷰 많은 책의 영향력이다.

  • 책 단위 — 각 책의 평균을 또 평균낸다. 리뷰가 100개든 5개든 책 한 권당 한 표.
  • 리뷰 단위 — 전체 리뷰를 합쳐 평균낸다. 리뷰 한 개당 한 표. 많이 읽힌 책이 기준선을 자기 쪽으로 끌어당긴다.

예: A(리뷰 100·평균 3.0), B(리뷰 5·평균 5.0), C책(리뷰 5·평균 5.0) → 책 단위는 4.33, 리뷰 단위는 3.18. 리뷰 수가 책마다 들쭉날쭉할 때 둘이 확 벌어진다.

Decision · C 산정 방식

리뷰 단위 가중평균을 쓴다

근거
지표는 본질적으로 여론의 반영이다. 100명이 읽고 매긴 평이 5명이 매긴 평보다 서비스 분위기를 더 잘 대표한다. IMDb도 투표 단위를 쓴다.
구현
이미 가진 집계 데이터(책별 AVG·COUNT)로 바로 계산. 쿼리 추가·DB 왕복 없음.

전체 C 하나 vs 장르별 C

Decision · C 범위

전체 하나로 둔다

근거
현재 표본이 적다. 장르별로 쪼개면 장르당 책이 몇 권 안 돼 각 C가 불안정해진다. 이론적 정교함보다 현재 규모에 맞는 선택.
유보
데이터가 쌓이고 장르별 평점 분포가 뚜렷이 갈리면 그때 장르별 C로 재검토.
03

설계 판단 기록

오늘 내린 결정들 — 각각이 면접에서 "왜?"에 답할 재료다

DR-01 · 계산 시점

Top10 계산을 조회 시점 → 집계 시점으로

문제
Rank 테이블은 실시간 계산을 피하려는 캐시인데, 정작 Top10을 조회(getRank) 때 계산하고 있었다. 테이블을 만든 취지와 모순.
결정
계산은 전부 새벽 배치에서 끝내고 조회는 순수하게 읽기만.
DR-02 · 책임 분배

집계는 DB, 가중 평점 계산은 애플리케이션

근거
집계(AVG·COUNT·필터)는 데이터가 많아 DB 옆에서 하는 게 빠르다. 가중 평점은 입력이 이미 작고(집계 끝난 수백 row) 도메인 정책이라 자바에 둔다.
핵심
JPA가 약해서가 아니다 — native면 윈도우 함수로 가능. 성격상 비즈니스 로직이고 입력이 작아서 앱에 두는 것.
DR-03 · 배치 후보 객체

집계 결과는 Object[] 대신 BookRatingStats로 받는다

근거
Object[]는 타입 안정성 없고(row[4] 캐스팅), 의미 없고, SELECT 순서에 묶인다. JPQL 생성자 표현식으로 의미 있는 객체에 담는다.
구분
집계 재료(BookRatingStats) ≠ 완제품(Rank 엔티티). 가중 평점·정렬 끝난 게 Rank. 둘을 섞으면 반쯤 빈 객체가 돌아다닌다.
DR-04 · 객체 형태

계산기는 static, 후보 객체는 record(불변)

static
WeightedScoreCalculator는 상태가 없는 순수 계산이라 인스턴스를 만들 이유가 없다. 빈으로 둘 적극적 이유(주입·무거운 생성)도 없음.
record
집계된 사실은 바뀌면 안 되는 불변 데이터. @Setter 빼고 record로.
DR-05 · 예외 설계

빈 리스트: 배치는 종료, 계산기는 예외

배치
집계 결과가 비면 = "아직 데이터 없음" = 정상. 로그 남기고 일찍 return.
계산기
그런데도 빈 게 오면 = 배치가 막았어야 할 사전조건 위반 = 비정상 → IllegalArgumentException.
통찰
같은 빈 리스트라도 받는 쪽에 따라 정상/비정상이 갈린다. 예외는 데이터로 정하는 게 아니라 그 메서드 입장의 정상 여부로 정한다.
double 함정
double의 0 나누기는 예외가 아니라 NaN이다. 정수 0 나누기는 ArithmeticException으로 즉시 터지지만, 0.0/0.0은 조용히 NaN이 나와 그 뒤 계산·정렬·저장까지 오염시킨다. "터지지 않고 망가지는" 케이스라 가드(fail fast)로 막는다.
DR-06 · 안 한 결정

전략 패턴은 도입하지 않는다 (YAGNI)

근거
점수 방식이 가중 평점 하나뿐이고, 런타임에 바뀔 축도 없고, if-else 지옥도 없다. 계층(추상화)을 두는 세 기준 중 어느 것도 충족 못 함.
격언
Rule of three — 비슷한 게 세 번째 나타날 때 추출한다. 미리 만든 추상화는 대개 잘못된 모양이 된다. 패턴을 안 쓸 줄 아는 게 더 드문 능력.
04

테스트 개념 정리

SI에서 몸에 밴 "전부 @SpringBootTest" 습관을 교정하는 핵심들

테스트는 3층

  • 단위 — 컨텍스트 없이 순수 객체. 의존성은 Mockito로 가짜 주입. 밀리초. 가장 많아야 함.
  • 슬라이스 — 필요한 레이어만. @DataJpaTest(쿼리·매핑), @WebMvcTest(컨트롤러·직렬화).
  • 통합 — @SpringBootTest 전체 컨텍스트. 느림. 진짜 중요한 시나리오만.

테스트 종류는 "로직이 어디 사는지"로 결정된다. 가중 평점은 자바 순수 계산이라 단위 테스트. 회독수·리뷰3개 필터는 쿼리에 있으니 @DataJpaTest로 실제 쿼리를 돌려야 검증된다.

교정
순수 계산에 @SpringBootTest + @Autowired를 쓰면 — 계산기 하나 두드리는데 사무실 전체에 불 켜는 격. 의존성 없는 객체는 new로 만들어 바로 호출하면 된다.

given · when · then

테스트 하나는 세 토막. given(상황 = 입력값 정하기) → when(검증할 동작 실행) → then(기대 결과 단언). 행동 목록을 먼저 글로 뽑아두면 그 문장 하나가 테스트 하나로 변환된다 — 생각과 코딩을 분리.

값 고르는 감각 — 대칭값 함정

v=3, m=3이면 비율이 정확히 0.5/0.5라 비율을 반대로 짠 버그도 4.5로 똑같이 통과한다. 좋은 테스트 값은 틀렸을 때 결과가 확 달라지는 비대칭 값(v=1, m=3 → 4.25). 경계값은 항상 딱 되는 값·딱 안 되는 값 양쪽을 찌른다(리뷰 3개 포함 / 2개 제외).

소수 비교 · 예외 검증

// 소수는 정확히 같은지(==, isEqualTo, Double.compare) 묻지 않는다
assertThat(score).isCloseTo(4.5, within(0.0001));

// 예외는 실행을 람다로 미뤄서 잡는다
assertThatThrownBy(() -> calc.calculateGlobalAverage(empty))
    .isInstanceOf(IllegalArgumentException.class);

assertTrue(조건)보다 assertThat(값) 계열이 낫다 — 터졌을 때 "expected 4.5 but was 0.0"처럼 실제값을 찍어줘 디버깅이 빠르다. 기대값은 반올림(3.18)하지 말고 식 그대로(350.0/110.0) 두는 게 안전하고 의도가 보인다.

05

행동 목록 (테스트 체크리스트)

랭킹 산정이 지켜야 할 규칙들 — 머릿속이 아니라 글로 뽑아 누락을 막는다

  • 장르별로 집계된다 / 한 장르 책은 다른 장르 랭킹에 섞이지 않는다 @DataJpaTest 또는 배치 테스트 · 두 규칙 한 시나리오로 묶기 가능
  • 장르가 없는 책은 집계에서 제외된다쿼리 조건
  • 리뷰가 3개인 책은 포함 / 2개인 책은 제외@DataJpaTest · HAVING COUNT>=3 경계 양쪽
  • 1회독 리뷰만 집계, 2회독 이상 제외@DataJpaTest · WHERE readingCount=1
  • 리뷰 적고 평점 높은 책 < 리뷰 많고 평점 약간 낮은 책단위 테스트 · 가중 평점 작동 = shrinkage 검증
  • 한 장르에 후보 11개여도 상위 10개만배치 테스트
  • 결과는 가중 평점 내림차순 정렬배치 테스트 · "10개 뽑기"와 "순서 세우기"는 다른 동작
체크 = 완료
칠해진 항목은 오늘 단위 테스트로 검증한 것(가중 평점 계산·C 계산). 빈 칸은 쿼리/배치 테스트로 남은 것 — 로드맵 문서 참고.

07

면접 한 줄 답변

각 결정을 "왜?"에 한 문장으로 — 이게 코드보다 중요하다

왜 가중 평점인가
단순 평균은 표본이 적은 책이 과대평가되는 문제가 있어, 리뷰 수를 반영한 가중 평점으로 정렬했습니다.
계산을 왜 배치로 옮겼나
Rank 테이블은 실시간 계산을 피하려는 캐시인데 Top10을 조회 시점에 계산하면 취지에 어긋나서, 계산을 전부 배치로 옮겼습니다.
DB와 앱의 역할 분배
집계는 데이터가 많아 DB에 맡기고, 가중 평점은 입력이 이미 작고 도메인 정책이라 애플리케이션으로 분리했습니다.
C 정책
지표는 본질적으로 여론의 반영이라 리뷰 단위 가중평균을 택했고, 표본이 적은 현재 규모에선 장르별 C가 불안정해 전체 하나로 뒀습니다.
예외 설계
빈 리스트는 배치 입장에선 정상이지만 계산기 입장에선 사전조건 위반이라, 배치는 일찍 종료하고 계산기는 fail fast로 예외를 던지게 했습니다.
double 0 나누기
double의 0 나누기는 예외가 아니라 NaN이라 조용히 전파되는 게 더 위험해서, 명시적으로 가드를 넣었습니다.
전략 패턴을 안 쓴 이유
점수 방식이 하나뿐이고 런타임에 바뀔 축도 없어서, 전략 패턴은 YAGNI로 판단해 도입하지 않았습니다.
테스트 타입 구분
순수 계산은 컨텍스트 없이 단위 테스트로, 쿼리 검증이 필요한 부분은 @DataJpaTest로 나눴습니다.
AI 도구 활용
테스트는 일부 AI 도구로 빠르게 작성하되, 무엇을 검증할지 고르고 짜인 테스트가 진짜 맞는 검증인지 리뷰하는 건 직접 합니다.
