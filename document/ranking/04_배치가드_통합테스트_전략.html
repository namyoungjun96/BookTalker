BookTalker · 결정 기록 04

배치 가드와 통합 테스트 전략

배치 맨 앞에 "빈 결과면 종료" 가드를 넣고, 그것을 실제 DB로 검증하는 통합 테스트를 처음 세웠다. 행위 검증 대신 상태 검증을 택한 이유, 트랜잭션·롤백의 함정, 그리고 이 테스트가 왜 "반쪽"인지를 정리한다.

이번 세션 · 완료
  • 배치 가드 구현 — 집계 결과가 비면 로그 + 종료 (deleteAll보다 앞)
  • 쿼리 단 테스트 마무리 — 회독수(규칙 A) / 리뷰 3개 경계(규칙 B) 각각 검증
  • 장르 분리는 쿼리 단 아님 확인 → 배치 단으로 이동
  • 배치 테스트 전략 확정 — 실제 DB 통합 테스트(@SpringBootTest), 상태 검증
  • 빈 결과 가드 통합 테스트 작성 (별도 클래스, @AfterEach 수동 정리)
01

가드의 위치가 곧 안전장치

"빈 결과면 종료"는 단순 return이 아니라 데이터 보호다

집계 결과가 비었을 때 만약 deleteAll을 먼저 하고 종료하면 — 어제 만든 멀쩡한 랭킹이 다 지워진 뒤 "오늘 집계할 게 없네" 하고 끝나서 랭킹 테이블이 텅 빈다. 사용자는 빈 화면을 본다. 그래서 가드는 반드시 deleteAll보다 앞, 실제 로직이 돌기 전에 위치한다.

List<BookRatingStats> stats = reviewRepository.aggregateRankData();
if (stats.isEmpty()) { log.info("집계 대상 없음, 갱신 건너뜀"); return; }  // 가드
// ↓ 여기부터 C 계산 · 가중평점 · 정렬 · Top10 · deleteAll · saveAll
이중 안전망

가드 + @Transactional

가드가 "빈 결과면 아예 시작 안 함"을 막고, @Transactional이 "중간에 터지면 deleteAll까지 롤백"을 보장한다. 둘이 합쳐져 "전부 되거나 전부 안 되거나"(원자성)가 지켜진다. 면접 "배치 중간 실패 시?" → 이 두 가지로 답.

02

상태 검증 vs 행위 검증

"deleteAll이 호출됐나"가 아니라 "기존 랭킹이 살아남았나"

✗ 행위 검증

verify(repo, never()).deleteAll()

"어떻게 동작하는지(내부 구현)"를 테스트에 박는다. 나중에 deleteAll → upsert로 구현만 바꿔도 결과는 멀쩡한데 테스트가 깨진다. 리팩토링의 발목.

✓ 상태 검증

기존 Rank를 넣어두고 → 배치 실행 → 그대로 남았나

"무엇을 하는가(결과)"를 본다. 구현이 deleteAll이든 upsert든, "빈 결과면 보존된다"는 결과만 맞으면 통과. 튼튼하다.

가드로 지키려던 진짜 가치는 "사용자가 빈 랭킹을 보지 않는 것"이었다. 그 가치를 랭킹이 살아있나로 직접 확인하는 게 상태 검증이다. deleteAll 호출 여부는 간접 증거일 뿐.

03

@SpringBootTest와 트랜잭션 함정

통합 테스트는 "진짜 커밋"을 봐야 하므로 자동 롤백을 일부러 안 쓴다

왜 @SpringBootTest인가 — 배치는 쿼리·계산기·저장을 조율해 최종 DB 상태를 만드는 게 핵심이다. 그 최종 상태가 맞는지 보려면 실제 빈을 다 띄워 진짜로 돌려야 한다. 개별 계산은 이미 단위 테스트(WeightedScoreCalculatorTest)로 분리했으니, 배치 테스트는 "합쳐져서 제대로 저장되나"만 본다.

롤백 함정
@SpringBootTest에 @Transactional을 붙이면 자동 롤백은 되지만 — 배치의 deleteAll·saveAll이 같은 트랜잭션이라 진짜 커밋을 못 본다. 영속성 컨텍스트 캐시인지 진짜 DB 상태인지 흐려진다. 그래서 배치 통합 테스트는 @Transactional을 안 붙이고, 대신 자동 롤백이 없으니 @AfterEach로 데이터를 직접 비운다.
잔실수

TestEntityManager는 @DataJpaTest 전용

TestEntityManager를 @SpringBootTest에 넣으면 주입 실패로 컨텍스트가 깨질 수 있다. @SpringBootTest에선 그냥 repository.save()(또는 saveAndFlush())로 저장. em.flush()/clear()도 불필요.

클래스 분리

단위와 통합은 같은 클래스에 안 섞는다

@SpringBootTest는 클래스 단위로 붙어 모든 테스트가 무거운 컨텍스트를 띄운다. 순수 단위 테스트(계산기)는 이미 별도 클래스로 빠져 있음. RankService에 통합 테스트만 있다면 RankServiceIntegrationTest(또는 IT) 접미사로 의도를 드러내는 게 관례 — 다만 지금처럼 통합밖에 없으면 분리는 취향(YAGNI 관점에선 미뤄도 됨).

04

이 테스트는 "반쪽"이다

빈 결과만 검증한다 — 짝이 있어야 가드가 완전히 증명된다

지금 가드 테스트는 "빈 결과 → 보존" 한 방향만 본다. 누가 실수로 aggregateRank 맨 앞에 if(true) return;을 박아 항상 종료하게 만들어도 이 테스트는 통과한다(빈 결과처럼 보존되니까).

완전해지려면 짝이 필요하다 — "데이터 있음 → 기존 지우고 새 랭킹으로 갱신". 이게 있어야 가드가 "빈 때만" 작동함이 증명된다. 그리고 이 짝은 곧 만들 배치 흐름 전체 테스트(Review 넣고 → 배치 → 가중평점 정렬되어 저장됐나)와 같다. 따로 만들 필요 없이 자연스럽게 채워진다.

하나의 테스트는 하나의 방향만 증명한다. 가드 같은 분기는 양쪽(작동할 때 / 안 할 때)이 다 잡혀야 완전하다. 지금은 한쪽, 나머지는 배치 흐름 테스트가 채운다.
검증 슬림하게
보존을 보는 테스트는 hasSize(1) + isbn 정도면 충분. 모든 필드(weightedScore·avgRating·title…)를 다 확인하면 "보존을 본다"는 의도가 흐려진다. 필드 정확성은 "데이터 있을 때 저장" 테스트의 몫.
05

다음 · 배치 흐름 테스트

구현이 아직 없는 부분을 만들고, 그 결과를 통합 테스트로

  • 배치 본체 구현 — 흐름 1~7 중 C 계산·가중평점·정렬·Top10·변환·저장 (가드는 완료)계산기 두 개는 이미 있음 · 조립만
  • 가중 평점 순 정렬 검증 — 저장된 Rank가 weightedScore 내림차순인가이게 04의 "반쪽" 가드 테스트의 짝
  • 장르별 Top10 컷 — 한 장르에 11개여도 10개만 저장배치 단 · groupingBy + limit(10)
  • 장르 분리 — 장르별로 올바르게 묶여 저장되나쿼리 단에서 배치 단으로 이동한 항목
  • weightedScore 값 정확성 — 저장된 점수가 계산식과 일치avgRating은 표시용, weightedScore는 정렬용 둘 다 저장 확인
시작점
다음 세션은 배치 본체 구현부터. 흐름은 결정 기록 03에 1~7로 확정돼 있고, 전략(@SpringBootTest, 트랜잭션 안 씀, @AfterEach 정리)은 이번에 정했다. 구현을 얹고 위 항목을 통합 테스트로 검증하면 된다.
