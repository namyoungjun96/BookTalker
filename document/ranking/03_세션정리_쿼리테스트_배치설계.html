BookTalker · 결정 기록 03

쿼리 · 테스트 · 배치 흐름 설계

이번 세션엔 집계 쿼리를 의미 있는 객체로 바꾸고, 첫 @DataJpaTest로 실제 DB에서 쿼리를 검증했고, 가중 평점 배치의 전체 흐름을 확정했다. 새로 부딪힌 것과 다음에 이어갈 지점을 정리한다.

이번 세션 · 완료
  • Rank에 weightedScore 컬럼 추가 + 타입 일관성 정리(double/int) · prod는 배포 전 ALTER 필요
  • 집계 쿼리 Object[] → BookRatingStats 생성자 표현식 + CAST(COUNT(r) AS Integer)
  • 첫 @DataJpaTest 세팅 — 실제 PostgreSQL(도커)로, H2 대체 끔
  • 쿼리 테스트 2개 — 리뷰 3개 집계+평균 검증 / 2개 제외 (경계 양쪽)
  • 배치 전체 흐름 확정 (집계 → C → 가중평점 → 정렬 → Top10 → 저장)
  • 조회단 문제 발견 — getRank(미사용) · getRankByGenre(조회 시점 계산) 둘 다 정리 대상
01

새로 부딪힌 것 · 배운 것

@DataJpaTest와 생성자 표현식에서 실제로 막혔던 지점들

@DataJpaTest란

필요한 만큼만 스프링을 띄운다

순수 계산은 컨텍스트 없이 단위 테스트. 하지만 쿼리 자체가 검증 대상일 땐(회독1·리뷰3개 필터가 진짜 거르나) 실제 DB에 데이터를 넣고 쿼리를 돌려야 한다 — Mockito로는 못 잡는다.

@DataJpaTest는 JPA 빈만 띄우고(가벼움), 각 테스트를 트랜잭션으로 감싸 끝나면 자동 롤백 → 테스트끼리 데이터가 안 섞인다. 여기선 @Autowired가 다시 맞다 — Repository는 스프링/JPA가 조립해줘야 동작하니까.

테스트 DB 선택

H2 대신 진짜 PostgreSQL

@DataJpaTest는 기본적으로 H2로 갈아끼우려 한다. 하지만 CAST·생성자 표현식 매핑은 방언 의존적이라, H2에서 통과해도 운영 PostgreSQL에선 다를 수 있다. @AutoConfigureTestDatabase(replace = NONE)로 운영과 같은 DB를 쓴다.

생성자 표현식

SELECT new {풀경로} + COUNT의 타입 함정

SELECT new com.example...dto.BookRatingStats(
  r.book.isbn13, r.book.genre, r.book.title, r.book.cover,
  AVG(r.rating), CAST(COUNT(r) AS Integer))
FROM Review r WHERE r.readingCount = 1
GROUP BY ... HAVING COUNT(r) >= 3

new와 풀 패키지 경로가 있어야 JPA가 객체로 만든다. 그리고 COUNT는 Long을 반환하는데 record의 reviewCount는 int라 "Missing constructor" 에러 → CAST(COUNT(r) AS Integer)로 맞췄다. (객체는 도메인이 원하는 int로 두고, 변환은 쿼리에서 — DB 반환 타입에 객체가 끌려가지 않는다.)

잔실수 모음
JPQL 문자열 이어붙일 때 줄 끝 공백 빠지면 COUNT(r))FROM으로 붙어 문법 에러 · @GeneratedValue id는 직접 넣지 말고 DB가 채우게 · hasSize()는 컬렉션용이라 result.size()(숫자)엔 못 씀 → assertThat(result).hasSize(1) 또는 .size()).isEqualTo(1)
쓰기 지연

save 후 em.flush()

JPA는 save해도 INSERT를 미룰 수 있다(쓰기 지연). 그 상태로 쿼리를 날리면 데이터를 못 봐 결과가 빈다. em.flush()로 DB에 강제 반영, em.clear()로 캐시를 비워 쿼리가 진짜 DB를 읽게 한다. "어쩌다 통과"가 아니라 통제된 통과로.

검증 강도

size만 보지 말고 값까지

hasSize(1)만 보면 리뷰 수가 틀려도, 평균이 깨져도, CAST가 값을 망쳐도 통과한다. "동작했나"가 아니라 "올바른 결과인가"를 봐야 안전망이 된다 → reviewCount()==3, avgRating().isCloseTo(4.0), 그리고 제외 케이스는 isbn으로 "남은 게 진짜 그 책"인지 직접 확인.

정렬의 성질

전체 정렬 → 그룹핑해도 장르 내 순서는 보존

정렬은 "전체 등수"가 아니라 상대 비교다. 소설 A가 B보다 높으면, 전체에서 정렬하든 소설끼리 정렬하든 A가 앞인 건 안 변한다. 중간에 다른 장르가 빠져도 등수만 당겨질 뿐 순서는 그대로. 그래서 전체 정렬 → 장르 그룹핑 → 각 10개(방식 A)로 충분하다 — 정렬 1회, 기존 패턴 재사용. 단, 정렬이 그룹핑보다 반드시 먼저여야 "상위"가 보장된다.

4·5 단계 구조
가중 평점 계산과 Rank 변환은 다른 단계로 끊는다(섞으면 책임이 엉켜 테스트가 못 가른다). 계산·정렬은 재료(BookRatingStats) 단계에서 끝내고, Rank는 마지막에 변환 — "재료 ≠ 완제품" 원칙. 정렬하려면 점수를 들고 있어야 하니 weightedScore를 일단 BookRatingStats에 추가. 재료에 가공물 섞이는 게 거슬리면 나중에 ScoredBook 별도 객체로 분리(지금은 YAGNI).
03

발견한 정리 대상 · 조회단

"안 쓰는 줄 알았더니" — 둘 다 조회 시점에 계산하는 같은 병

getRank(genre)미사용
  • 엔드포인트·호출처 모두 없음 — 죽은 코드
  • 정렬이 AvgRating 기준 (가중 평점 아님)
  • findTop10By... — 조회 시점에 Top10 자름
getRankByGenre()실제 사용
  • findAllGroupable()로 전체를 끌어와 stream groupingBy → limit(10) — 매 조회마다 그룹핑·자르기 곡예
  • 정렬 보장이 불분명 — 정렬 안 된 채 limit이면 "상위 10"이 아니라 "아무 10"

두 메서드 다 Rank 테이블을 캐시로 둔 취지를 무너뜨린다. 배치(다음 단계)가 정렬·Top10을 미리 끝내 저장하면, 조회는 "장르 필터 + weightedScore 순 읽기"로 단순해지고 이 곡예가 사라진다.

순서
조회단 정리는 배치 구현 다음. 배치가 weightedScore를 채워줘야 그 기준으로 정렬·검증할 수 있다. 지금 조회부터 고치면 정렬할 값이 없다.
04

다음에 할 일 · 테스트 시나리오 맵

행동 목록을 "어디서 검증할지"로 분류하면 그게 곧 작업 순서

쿼리 단 @DataJpaTest
  • 리뷰 3개는 집계 + 평균 검증
  • 리뷰 2개는 제외 (경계 반대쪽)
  • 1회독만 집계 / 2회독 이상 제외
  • 장르가 다르면 다른 그룹으로 분리
계산 단 단위 테스트
  • 가중 평점 정상 계산
  • 리뷰 적으면 전체평균쪽으로 (shrinkage)
  • C 정상 계산 / 빈 리스트면 예외
배치 단 새로 나올 시나리오
  • 결과가 가중 평점 내림차순 정렬된다
  • 한 장르에 11개여도 상위 10개만 저장
  • 빈 결과면 저장하지 않고 종료
  • 저장된 Rank의 weightedScore가 맞다
엣지케이스는 무한하지 않다. 핵심은 경계 — 리뷰 3개/2개, Top10의 10개/11개, 빈 리스트. 경계만 양쪽으로 찌르면 대부분 잡힌다. "모든 경우의 수"가 아니라 "버그가 숨는 경계"만 골라 친다.
